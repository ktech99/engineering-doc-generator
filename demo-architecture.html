NexaCommerce

Microservices e-commerce platform — Node.js API Gateway + Go domain services + PostgreSQL + Redis. Five independently deployable services communicating over gRPC with an event bus for async workflows.

API Gateway

Single ingress point. JWT auth, rate limiting, request routing to domain services.

⚡

Event-Driven Core

NATS JetStream for async workflows. Order placed → payment → fulfillment pipeline.

🔒

Zero-Trust Auth

Auth Service issues short-lived JWTs. Services validate locally — no central auth hop.

Info

All services are containerized and deployed via Helm. Each service owns its own PostgreSQL schema; no cross-service DB queries.

Design choice

Each service owns its schema. Cross-service data is accessed through events or explicit API calls — never shared table joins. This lets services scale and deploy independently.

Convention

Services import nothing from each other's internal/ packages. Cross-boundary contracts are defined only in packages/proto and packages/events.

Components

What each service does, why it exists, and where to find it.

Gateway API Gateway
Single HTTP/2 ingress for all client traffic. Validates JWTs (local JWKS), enforces rate limits (Redis token bucket), and proxies requests to domain services via gRPC. Does NOT contain business logic — it's a dumb router with auth enforcement.
Node.jsExpressJWTRate Limit
Auth Auth Service
Issues and rotates JWTs using RS256. Maintains a JWKS endpoint for gateways to pull public keys. Sessions stored in Redis with a 24h TTL. Refresh token rotation prevents replay. OAuth2 PKCE flow for third-party logins.
GoRS256 JWTRedis
Orders Order Service
Owns the order lifecycle — draft → confirmed → paid → shipped → delivered. Each state transition is validated, persisted, and published as a CloudEvent to NATS JetStream. Idempotent order creation via client-generated UUIDs.
GoPostgreSQLNATS
Payments Payment Service
Consumes OrderPlaced events and initiates Stripe PaymentIntents. Emits PaymentSettled or PaymentFailed. Handles Stripe webhook retries with exactly-once guarantees using a dedupe table.
GoStripe APINATS
Fulfillment Fulfillment Service
Listens for PaymentSettled, creates pick-pack-ship jobs, and calls the warehouse WMS via REST. Exposes a webhook endpoint for carrier tracking updates. Owns shipment and tracking data.
GoWMS RESTNATS

Order interface (TypeScript)

// packages/proto/order.ts — shared contract between gateway and order-service

export interface CreateOrderRequest {
  idempotencyKey: string;       // UUID v4, client-generated
  customerId:     string;       // from JWT sub claim
  items: OrderItem[];
  shippingAddress: Address;
  currency: "USD" | "EUR" | "GBP";
}

export interface OrderItem {
  sku:       string;
  quantity:  number;            // min: 1
  unitPrice: number;            // cents
}

export interface OrderResponse {
  id:        string;            // UUID
  status:    OrderStatus;
  total:     number;            // cents
  createdAt: string;            // ISO 8601
  items:     OrderItem[];
}

export type OrderStatus =
  | "draft" | "confirmed" | "paid"
  | "shipped" | "delivered" | "cancelled";

Data Model

Three PostgreSQL schemas — one per owning service. No foreign keys across schemas.

orders schema
orders — id (UUID PK), customer_id, status, currency, total_cents, idempotency_key, created_at, updated_at

order_items — id, order_id (FK), sku, quantity, unit_price_cents

order_events — append-only status transition log with actor + metadata
payments schema
payments — id, order_id, stripe_payment_intent_id, amount_cents, status, settled_at

payment_events — Stripe webhook events (deduped by event_id)

refunds — refund_id, payment_id, amount_cents, reason, created_at
fulfillment schema
shipments — id, order_id, wms_job_id, carrier, tracking_number, status

tracking_events — carrier webhook events, timestamped location/status updates

delivery_proofs — image URLs, signature blobs, delivered_at
Consistency model

Cross-service data consistency is eventual — driven by NATS events. A PaymentSettled event may arrive 50–200ms after the Stripe webhook. Services must tolerate brief inconsistency windows and implement idempotent consumers.

Runbook

How to run locally, promote to staging, and roll back a bad deploy.

Local dev

  1. Clone repo and install deps: git clone git@github.com:nexacommerce/platform.git && cd platform

  2. Start infra: docker compose up -d — starts Postgres, Redis, NATS

  3. Seed databases: ./scripts/seed-db.sh

  4. Start all services: ./scripts/run-local.sh — watches for changes, auto-restarts

  5. Hit the gateway at http://localhost:3000

Rollback

Rollback

Each service is deployed independently. To roll back a single service: helm rollback <service-name> -n production. DB migrations are backward-compatible — old code runs against new schema safely.
