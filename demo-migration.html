PostgreSQL → Aurora Migration

Migrating the NexaCommerce order, payment, and fulfillment schemas from self-managed PostgreSQL 14 on RDS to Amazon Aurora PostgreSQL 15. Zero planned downtime. Live traffic cutover using logical replication + feature flag. 3 phases over 6 weeks.

📈 Why migrate

Aurora scales reads horizontally (up to 15 replicas), has automated failover in <30s, and costs 20% less at our current storage volume.

🔄 How we cut over

Logical replication keeps Aurora in sync with RDS. Feature flag flips writes to Aurora. RDS becomes read-only standby for 48h.

🛡️ Safety net

RDS instance kept alive for 48h post-cutover. Rollback: flip feature flag back, Aurora writes drain, RDS resumes primary role.

Scope

This plan covers the orders, payments, and fulfillment schemas. The auth schema (users, sessions) migrates separately in Q3 — different risk profile, different owner.

Phases

Three phases over six weeks. Phase 1 is infra-only. Phase 2 is the live replication window. Phase 3 is cutover and cleanup.

Phase 1 Provision Aurora · Weeks 1–2
Terraform new Aurora cluster in same VPC. Run schema validation and row-count reconciliation against a snapshot restore. Instrument Aurora metrics in Datadog. No traffic change yet.
Phase 2 Logical Replication · Weeks 3–4
Enable PostgreSQL logical replication from RDS → Aurora. Replicate all 14 tables. Verify lag stays below 500ms at peak load. Shadow-read 5% of queries to Aurora, compare results. Iterate on any discrepancies.
Phase 3 Cutover · Week 5–6
Feature flag flips writes to Aurora during a low-traffic window (Sunday 03:00 UTC). Confirm replication lag reaches zero. Monitor for 48h with RDS as hot standby. Decommission RDS after bake period.
Note

The 48h RDS hot standby is non-negotiable. It's the primary rollback mechanism. Decommission only after two full business-day cycles with Aurora as primary.

Migration Steps

Ordered execution steps. Each step has an owner and a verification gate before proceeding.

  1. Provision Aurora cluster via Terraform

    Run terraform apply -target=module.aurora. Cluster: db.r6g.large, PostgreSQL 15.4, Multi-AZ, gp3 storage. Verify cluster endpoint accessible from service VPC.

  2. Restore snapshot and validate schema

    Restore latest RDS snapshot into Aurora. Run ./scripts/validate-schema.sh — compares table DDL, indexes, constraints, row counts. Must show 0 diffs before proceeding.

  3. Enable logical replication on RDS

    Set rds.logical_replication = 1 in parameter group. Requires RDS minor restart (~90s). Create publication: CREATE PUBLICATION nexacomm_pub FOR ALL TABLES;

  4. Create subscription on Aurora

    On Aurora: CREATE SUBSCRIPTION nexacomm_sub CONNECTION '...' PUBLICATION nexacomm_pub;. Initial table sync will begin. Monitor pg_stat_subscription until srsubstate = 'r' (streaming) for all tables.

  5. Verify replication lag < 500ms at peak

    Run load test (k6 run scripts/peak-load.js). Monitor pg_replication_slots.confirmed_flush_lsn. Aurora must stay within 500ms of RDS during 3× baseline load for 30 continuous minutes.

  6. Shadow-read 5% of queries to Aurora

    Enable feature flag AURORA_SHADOW_READ_PCT=5. Service reads from Aurora, compares result with RDS response, logs any discrepancy to Datadog. Run for 1 week. Fix any mismatches before proceeding.

  7. Cutover: flip writes to Aurora

    Sunday 03:00 UTC. Set AURORA_PRIMARY=true via LaunchDarkly — gradual 0%→5%→25%→100% rollout over 20 minutes. Watch error rate and latency. RDS becomes read-only.

  8. 48h bake — monitor and validate

    Aurora as primary for two full business days. Monitor: P50/P99 query latency, error rate, row count reconciliation. RDS instance kept running as hot standby for instant rollback.

  9. Decommission RDS

    Take final RDS snapshot (retained 30 days). Run terraform destroy -target=module.rds_legacy. Update DNS CNAME for db.internal to Aurora writer endpoint. Remove AURORA_PRIMARY flag — Aurora is now the only DB.

Risks

Known risks, likelihood, impact, and mitigation. All risks reviewed and accepted by engineering lead and CTO.

Risk 1 Replication lag spike during peak
If RDS → Aurora lag exceeds 500ms at cutover, stale reads could cause order state mismatches. Mitigation: we abort cutover if lag >500ms. Lag gate is automated — script polls pg_replication_slots and blocks flag flip until clear.
Risk

Aurora PostgreSQL 15 has one known behavioral difference: RETURNING clauses on ON CONFLICT DO NOTHING silently return no rows (vs PG14 which returns the existing row). We've patched 3 call sites in the payments service that relied on this behavior. Test coverage added.

Risk 2 Stripe deduplication table gap
The stripe_dedupe table uses a sequence-backed ID. During cutover, if sequences on Aurora and RDS diverge, a Stripe retry could bypass deduplication. Mitigation: aurora sequences advanced +10k past RDS before cutover. Duplicate payment monitor alert set on Stripe dashboard for 72h post-cutover.
Risk 3 Aurora writer failover during bake window
Aurora auto-failover is <30s but causes a brief write pause. If this happens during the 48h bake, services see transient errors. Mitigation: all DB clients use exponential backoff with 3× retry. Tested during Phase 1 by forcing a failover on the Aurora test cluster.

Rollback Plan

Three rollback points, each with clear trigger criteria and estimated time to recovery.

Rollback A Before cutover (Phase 1–2)
Trigger: any schema validation mismatch or shadow-read discrepancy >0.01%. Action: drop Aurora subscription, destroy Aurora cluster via Terraform. RDS continues as-is. No user impact ever. Recovery: immediate.
Rollback B During cutover rollout (Phase 3, <30 min)
Trigger: error rate >0.5% or P99 latency >3s during flag rollout. Action: flip AURORA_PRIMARY=false via LaunchDarkly — all writes immediately return to RDS. Aurora writes to-date are discarded (lag was <30s). Recovery time: <2 minutes.
Rollback C During 48h bake window
Trigger: sustained error rate >0.1% or P99 >2s for >5 minutes. Action: flip flag back to RDS. Any Aurora-only writes (post-cutover) must be manually reconciled via DMS change-data-capture log. Recovery time: ~5 minutes for traffic, ~2h for full reconciliation. This path requires engineering lead approval.
Go/No-go

We will not proceed to Phase 3 cutover without explicit sign-off from: DB lead, on-call engineer, and engineering manager. A 30-minute pre-cutover call is mandatory. The cutover window is automatically aborted if replication lag >500ms at T-5 minutes.
