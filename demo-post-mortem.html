INC-2847 — Payment Service Outage

On 2026-04-14 at 02:17 UTC, a misconfigured Redis connection pool exhausted all available connections during a traffic surge, causing the Payment Service to stop processing transactions. 4 hours of degraded checkout. No data lost. Fixed same day.

🔥 What broke

Redis connection pool hit its 50-connection limit. Payment Service goroutines deadlocked waiting for a free slot.

🕵️ How we found it

P99 latency alert fired at 02:23 UTC. On-call engineer correlated Redis metrics within 8 minutes.

✅ How we fixed it

Raised pool limit to 200, deployed connection timeout guard, added Redis pool saturation alert.

Severity

SEV-2: Partial outage. Checkout was unavailable for ~68% of users (those hitting affected pods). Auth and Order browsing remained functional throughout.

Incident Timeline

All times UTC. Total incident window: 02:17 – 06:31 (4h 14m).

Figure 1 — Incident timeline. Detection lag: 6 minutes. Time to root cause: 53 minutes. Time to fix: 2h 28m.

Note

The pod restarts at 02:31 masked the symptoms temporarily (goroutines released) but did not address the root cause. This extended MTTR by ~30 minutes.

Root Causes

Three contributing factors — all required for the outage to occur. Any one fixed would have prevented it.

RC-1 Redis pool size not scaled with pod count
The connection pool cap was set to 50 during initial development (single pod). When HPA scaled the Payment Service to 8 pods under load, total Redis connections hit 400 — 8× what ElastiCache was configured to handle. New connections were rejected; goroutines waited indefinitely.
RC-2 No connection timeout on Redis client
redis.NewClient was initialized without a DialTimeout or ReadTimeout. Goroutines acquiring a pool slot blocked forever — not for a bounded retry period. This caused cascading queue growth.
RC-3 Missing saturation alert for Redis pool
We had latency alerts but no alert on Redis pool utilization. Pool hit 100% 6 minutes before the P99 alert fired. Earlier detection would have allowed a pre-emptive scale-out before user impact.
Contributing factor

A flash sale campaign launched at 01:55 UTC generated 3.2× normal traffic. The traffic spike was known but the Redis capacity implications were not analyzed in the pre-launch checklist.

What We Fixed

Immediate mitigations deployed during the incident, plus follow-up hardening completed within 48 hours.

Done

All three root causes have been addressed. The fixes were deployed to production at 04:45 UTC on 2026-04-14 and validated through the following business day peak.

Fix 1 Redis pool scaled to pod-aware limit
Pool size now set to maxPoolPerPod × maxPods = 25 × 10 = 250. Value injected via env var REDIS_POOL_SIZE, defaulting to 250. Helm values updated. ElastiCache max connections raised to 500 in Terraform.
Fix 2 Connection and read timeouts enforced
Redis client now initialized with DialTimeout: 500ms, ReadTimeout: 1s, WriteTimeout: 1s, and PoolTimeout: 2s. Goroutines now fail fast and return a 503 rather than blocking indefinitely.
Fix 3 Redis pool saturation alert added
New Datadog monitor: alert when redis.pool_hits / (redis.pool_hits + redis.pool_misses) < 0.85 for 2 consecutive minutes. PagerDuty integration configured. Alert tested against incident replay — would have fired 6 minutes before user impact.
Fix 4 Pre-launch capacity checklist updated
Added mandatory step: for any campaign expected to generate >2× baseline traffic, run ./scripts/capacity-check.sh which validates Redis pool headroom, DB connection limits, and HPA max replica counts against projected peak load.

Q&A

Questions raised in the incident review and all-hands. Answered by the on-call team.

Q. Why didn't restarting the pods fix the problem?
Pod restarts temporarily released goroutines blocked on pool acquisition, giving a brief reprieve. But the root cause — pool cap too low for the current replica count — was unchanged. Within 2–3 minutes of restart, the same traffic pattern re-exhausted the pool.
Q. Was any order or payment data lost or duplicated?
No. Requests that failed returned HTTP 503. No partial writes occurred. The Payment Service's idempotency-key deduplication table was intact. Stripe confirmed zero duplicate charges. All queued NATS events were replayed cleanly after recovery.
Q. Why did it take 53 minutes to identify Redis as the root cause?
Initial investigation focused on pod health and Kubernetes events (the pod restarts muddied the signal). The Redis pool metric wasn't on the default incident dashboard. The fix: Redis pool utilization is now in the first panel engineers see during a Payment Service incident.
Q. How many customers were affected?
Approximately 14,200 unique sessions experienced checkout failures between 02:17 and 06:31 UTC. 3,400 users received an error mid-checkout. CX team is following up with courtesy credits.
Blameless

This post-mortem is blameless. The pool configuration was set during a period when the service ran at 1 replica — a perfectly reasonable decision at the time. The system lacked the observability needed to detect drift as scale increased. That's the process failure, not a person failure.
